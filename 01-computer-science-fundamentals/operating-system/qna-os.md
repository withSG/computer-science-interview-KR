## Q1. 프로세스와 스레드의 차이점은 무엇인가요? ⭐⭐⭐

<details>
<summary>답변 보기</summary>

### 핵심 답변
프로세스는 **자원 할당의 단위**이고, 스레드는 **실행의 단위**입니다. 프로세스는 독립적인 메모리 공간을 가지며, 스레드는 같은 프로세스 내에서 Code, Data, Heap 영역을 공유하고 Stack만 독립적으로 가집니다.

### 상세 설명
- **프로세스**: 독립적인 메모리 공간 (Code, Data, Heap, Stack 모두 독립)
- **스레드**: 프로세스 내에서 Code, Data, Heap 공유, Stack만 독립

### 면접관이 주목하는 포인트
- 컨텍스트 스위칭 비용 차이 설명
- 메모리 공유로 인한 동기화 이슈 언급
- 실무 선택 기준 제시

### 꼬리 질문 대비
- "멀티 프로세스와 멀티 스레드 중 언제 무엇을 선택하나요?"
  → 안정성 필요: 멀티 프로세스, 성능 필요: 멀티 스레드
- "크롬 브라우저가 멀티 프로세스 방식을 사용하는 이유는?"
  → 탭 하나의 크래시가 전체에 영향을 주지 않도록 격리

### 관련 개념
- [프로세스와 스레드](./01-process-thread.md)

</details>

## Q2. 컨텍스트 스위칭이란 무엇이며, 성능에 어떤 영향을 미치나요? ⭐⭐

<details>
<summary>답변 보기</summary>

### 핵심 답변
컨텍스트 스위칭은 CPU가 현재 실행 중인 프로세스(스레드)의 상태를 저장하고, 다른 프로세스(스레드)의 상태를 복원하여 실행을 전환하는 과정입니다.

### 상세 설명

**저장되는 컨텍스트:**
- Program Counter (PC)
- CPU 레지스터
- 메모리 관리 정보
- I/O 상태 정보

**비용 차이:**
- 프로세스 스위칭: TLB 플러시, 캐시 미스 → 비용 높음
- 스레드 스위칭: 레지스터만 교체 → 비용 낮음

### 면접관이 주목하는 포인트
- TLB 플러시의 영향 설명
- 캐시 적중률 저하 언급
- 성능 최적화 방안 제시

### 꼬리 질문 대비
- "컨텍스트 스위칭 오버헤드를 줄이는 방법은?"
  → 스레드 풀 사용, 비동기 I/O, CPU 친화도 설정

### 관련 개념
- [컨텍스트 스위칭](./03-context-switching.md)

</details>

## Q3. 데드락(교착 상태)이란 무엇이며, 발생 조건 4가지를 설명해주세요. ⭐⭐⭐

<details>
<summary>답변 보기</summary>

### 핵심 답변
데드락은 두 개 이상의 프로세스(스레드)가 서로가 보유한 자원을 기다리며 무한 대기하는 상태입니다.

### 발생 조건 (4가지 모두 충족 시)
1. **상호 배제 (Mutual Exclusion)**: 자원은 한 번에 하나의 프로세스만 사용 가능
2. **점유 대기 (Hold and Wait)**: 자원을 보유한 채로 다른 자원을 기다림
3. **비선점 (No Preemption)**: 다른 프로세스의 자원을 강제로 빼앗을 수 없음
4. **순환 대기 (Circular Wait)**: 프로세스들이 순환 형태로 자원을 기다림

### 해결 방법
- **예방**: 4가지 조건 중 하나 제거
- **회피**: 은행원 알고리즘 (안전 상태 유지)
- **탐지/복구**: 주기적 탐지 후 프로세스 종료
- **무시**: 발생 확률 낮으면 무시 (시스템 재시작)

### 면접관이 주목하는 포인트
- 4가지 조건 정확히 암기
- 실무에서의 해결 경험
- 예방 vs 회피 vs 탐지 트레이드오프

### 꼬리 질문 대비
- "프로젝트에서 데드락을 경험한 적 있나요?"
- "순환 대기 조건을 깨는 방법은?"
  → 자원에 순서를 부여하고, 순서대로만 자원 요청

### 관련 개념
- [교착 상태와 경쟁 상태](./04-deadlock-race-condition.md)

</details>

## Q4. 경쟁 상태(Race Condition)란 무엇이며, 어떻게 해결하나요? ⭐⭐⭐

<details>
<summary>답변 보기</summary>

### 핵심 답변
경쟁 상태는 두 개 이상의 스레드가 공유 자원에 동시에 접근하여 데이터의 정합성이 깨지는 현상입니다.

### 해결 방법

| 방법 | 특징 |
|------|------|
| synchronized | 간단, 암묵적 락, 블로킹 |
| ReentrantLock | 명시적, tryLock/타임아웃 가능 |
| Atomic 클래스 | Lock-free, CAS 연산 |
| volatile | 가시성 보장 (원자성 X) |

### 상세 설명
```java
// 문제 상황
count++; // 원자적이지 않음 (read → modify → write)

// 해결 1: synchronized
public synchronized void increment() { count++; }

// 해결 2: Atomic
AtomicInteger count = new AtomicInteger();
count.incrementAndGet();
```

### 면접관이 주목하는 포인트
- 원자성(Atomicity) 개념 이해
- CAS(Compare-And-Swap) 연산 설명
- 실무 적용 경험

### 꼬리 질문 대비
- "CAS 연산이 무엇인가요?"
  → 현재 값을 예상 값과 비교 후 일치하면 새 값으로 교체하는 원자적 연산
- "synchronized와 ReentrantLock의 차이점은?"
  → ReentrantLock은 타임아웃, 인터럽트 가능, 공정성 설정 등 더 세밀한 제어 가능

### 관련 개념
- [교착 상태와 경쟁 상태](./04-deadlock-race-condition.md)

</details>

## Q5. Stack과 Heap 메모리의 차이점은 무엇인가요? ⭐⭐⭐

<details>
<summary>답변 보기</summary>

### 핵심 답변
Stack은 함수 호출과 지역 변수를 저장하는 영역으로 컴파일 타임에 크기가 결정되고 자동 관리됩니다. Heap은 동적으로 할당되는 객체를 저장하는 영역으로 런타임에 할당되고 GC 또는 수동으로 관리됩니다.

### 비교 표

| 구분 | Stack | Heap |
|------|-------|------|
| 할당 시점 | 컴파일 타임 | 런타임 |
| 크기 | 제한적 (~1MB) | 상대적으로 큼 |
| 속도 | 빠름 | 느림 |
| 관리 | 자동 | GC 또는 수동 |
| 스레드 공유 | 독립적 | 공유 |
| 에러 | StackOverflowError | OutOfMemoryError |

### 면접관이 주목하는 포인트
- 각 영역에 저장되는 데이터 유형
- 메모리 에러 종류와 원인
- GC와의 연관성

### 꼬리 질문 대비
- "StackOverflowError는 언제 발생하나요?"
  → 재귀 호출이 너무 깊거나, 지역 변수가 너무 많을 때
- "OutOfMemoryError 발생 시 어떻게 디버깅하나요?"
  → 힙 덤프 분석 도구(MAT, VisualVM) 사용

### 관련 개념
- [메모리 관리](./02-memory-management.md)

</details>

## Q6. 가상 메모리란 무엇이며 왜 필요한가요? ⭐⭐

<details>
<summary>답변 보기</summary>

### 핵심 답변
가상 메모리는 물리적 메모리 크기와 관계없이 프로세스에게 큰 주소 공간을 제공하는 기술입니다. 디스크를 메모리의 확장으로 사용하여 물리 메모리보다 큰 프로그램도 실행할 수 있게 합니다.

### 필요성
1. 물리 메모리보다 큰 프로그램 실행
2. 프로세스 간 메모리 격리 및 보호
3. 메모리 단편화 해결

### 핵심 개념
- **페이지**: 가상 메모리의 고정 크기 단위 (보통 4KB)
- **프레임**: 물리 메모리의 고정 크기 단위
- **페이지 테이블**: 가상 주소 → 물리 주소 매핑
- **TLB**: 주소 변환 캐시

### 면접관이 주목하는 포인트
- 페이지 폴트 처리 과정
- TLB의 역할과 컨텍스트 스위칭과의 관계
- 스래싱(Thrashing) 개념

### 꼬리 질문 대비
- "페이지 폴트가 발생하면 어떤 일이 일어나나요?"
  → 인터럽트 발생 → 디스크에서 페이지 로드 → 페이지 테이블 업데이트 → 재실행
- "스래싱이란 무엇인가요?"
  → 페이지 폴트가 과도하게 발생하여 CPU가 실제 작업보다 페이지 교체에 더 많은 시간을 소비하는 현상

### 관련 개념
- [가상 메모리](./05-virtual-memory.md)

</details>

## Q7. 뮤텍스(Mutex)와 세마포어(Semaphore)의 차이점은 무엇인가요? ⭐⭐

<details>
<summary>답변 보기</summary>

### 핵심 답변
뮤텍스는 **1개의 스레드만** 임계 영역에 접근할 수 있게 하는 동기화 도구이고, 세마포어는 **지정된 개수의 스레드**가 동시에 접근할 수 있게 하는 동기화 도구입니다.

### 비교

| 구분 | Mutex | Semaphore |
|------|-------|-----------|
| 동시 접근 | 1개 | N개 (count 지정) |
| 소유권 | 있음 (락 건 스레드만 해제) | 없음 |
| 사용 사례 | 상호 배제 | 리소스 풀 제한 |

### 예시
```java
// Semaphore 예시 (커넥션 풀)
Semaphore semaphore = new Semaphore(10); // 최대 10개

semaphore.acquire(); // 리소스 획득
try {
    // 커넥션 사용
} finally {
    semaphore.release(); // 리소스 반환
}
```

### 면접관이 주목하는 포인트
- 이진 세마포어와 뮤텍스의 차이
- 실무 적용 사례 (커넥션 풀, 스레드 풀)

### 꼬리 질문 대비
- "이진 세마포어와 뮤텍스의 차이점은?"
  → 뮤텍스는 소유권 개념이 있어 락을 건 스레드만 해제 가능, 이진 세마포어는 소유권 없음

</details>

## Q8. 페이지 교체 알고리즘 중 LRU에 대해 설명해주세요. ⭐

<details>
<summary>답변 보기</summary>

### 핵심 답변
LRU(Least Recently Used)는 가장 오래 사용되지 않은 페이지를 교체하는 알고리즘입니다. 최근에 사용된 페이지는 가까운 미래에도 사용될 가능성이 높다는 지역성 원리에 기반합니다.

### 페이지 교체 알고리즘 비교

| 알고리즘 | 설명 | 장점 | 단점 |
|---------|------|------|------|
| FIFO | 먼저 들어온 페이지 교체 | 구현 간단 | Belady's Anomaly |
| LRU | 가장 오래 미사용된 페이지 교체 | 성능 좋음 | 구현 복잡 |
| LFU | 사용 빈도 낮은 페이지 교체 | 빈도 고려 | 최근성 무시 |

### 면접관이 주목하는 포인트
- LRU 구현 방법 (연결 리스트 + 해시맵)
- 시간 복잡도: O(1)

### 꼬리 질문 대비
- "LRU를 어떻게 구현하나요?"
  → HashMap + Doubly Linked List 조합으로 O(1) 구현

</details>

## Q9. 프로세스와 프로그램의 차이는 무엇인가요? ⭐⭐

<details>
<summary>답변 보기</summary>

### 핵심 답변

| 구분 | 프로그램 | 프로세스 |
|------|---------|---------|
| 상태 | 정적 (저장된 코드) | 동적 (실행 중인 코드) |
| 위치 | 디스크 | 메모리 |
| 자원 | 없음 | CPU, 메모리 할당 |
| 생명주기 | 영구적 | 생성~종료 |

### 상세 설명
- **프로그램**: 디스크에 저장된 실행 파일 (수동적)
- **프로세스**: 메모리에 로드되어 실행 중인 프로그램 인스턴스 (능동적)

### 예시
```
notepad.exe (프로그램) → 실행 → notepad 프로세스
같은 프로그램을 여러 번 실행하면 여러 프로세스 생성
```

### 면접관이 주목하는 포인트
- 프로세스가 가지는 자원 (PCB, 메모리 공간)
- 하나의 프로그램에서 여러 프로세스가 생성될 수 있음

</details>

## Q10. 인터럽트(Interrupt)란 무엇인가요? ⭐⭐

<details>
<summary>답변 보기</summary>

### 핵심 답변
인터럽트는 CPU가 현재 작업을 중단하고 **우선적으로 처리해야 할 이벤트**가 발생했음을 알리는 신호입니다.

### 인터럽트 종류

| 종류 | 설명 | 예시 |
|------|------|------|
| 하드웨어 인터럽트 | 외부 장치에서 발생 | 키보드, 마우스, 타이머 |
| 소프트웨어 인터럽트 | 프로그램 내부에서 발생 | 시스템 콜, 예외 |

### 인터럽트 처리 과정
```
1. 인터럽트 발생
2. 현재 상태 저장 (PC, 레지스터)
3. 인터럽트 벡터 테이블에서 핸들러 주소 조회
4. ISR (Interrupt Service Routine) 실행
5. 저장된 상태 복원
6. 원래 작업 계속
```

### 면접관이 주목하는 포인트
- 폴링 vs 인터럽트 방식 비교
- 인터럽트 우선순위

</details>

## Q11. 시스템 콜(System Call)이란 무엇인가요? ⭐⭐

<details>
<summary>답변 보기</summary>

### 핵심 답변
시스템 콜은 사용자 프로그램이 **커널의 기능을 요청**하기 위한 인터페이스입니다. 사용자 모드에서 커널 모드로 전환되어 실행됩니다.

### 시스템 콜 종류

| 분류 | 시스템 콜 예시 |
|------|--------------|
| 프로세스 제어 | fork(), exec(), exit(), wait() |
| 파일 관리 | open(), read(), write(), close() |
| 장치 관리 | ioctl(), read(), write() |
| 정보 유지 | getpid(), time(), alarm() |
| 통신 | pipe(), socket(), send(), recv() |

### 동작 과정
```
1. 사용자 프로그램에서 라이브러리 함수 호출 (printf)
2. 라이브러리가 시스템 콜 호출 (write)
3. 소프트웨어 인터럽트 발생 (trap)
4. 커널 모드로 전환
5. 커널에서 요청 처리
6. 사용자 모드로 복귀
```

### 면접관이 주목하는 포인트
- 왜 시스템 콜이 필요한지 (보안, 안정성)
- 함수 호출과 시스템 콜의 차이

</details>

## Q13. DMA(Direct Memory Access)란 무엇인가요? ⭐

<details>
<summary>답변 보기</summary>

### 핵심 답변
DMA는 CPU 개입 없이 **I/O 장치가 메모리에 직접 접근**하여 데이터를 전송하는 방식입니다.

### DMA가 필요한 이유
```
[기존 방식 - PIO (Programmed I/O)]
1. CPU가 I/O 장치에서 데이터 읽기
2. CPU가 메모리에 데이터 쓰기
3. 모든 데이터 전송에 CPU 사용
→ CPU가 I/O 작업에 묶임

[DMA 방식]
1. CPU가 DMA 컨트롤러에 전송 요청
2. DMA가 I/O → 메모리 직접 전송
3. 전송 완료 시 인터럽트로 CPU에 알림
→ CPU가 다른 작업 수행 가능
```

### DMA 동작 과정
1. CPU가 DMA 컨트롤러에 전송 정보 설정 (소스, 목적지, 크기)
2. DMA가 버스 제어권 획득
3. 데이터 블록 전송
4. 전송 완료 시 인터럽트

### 면접관이 주목하는 포인트
- CPU 효율성 향상
- 대용량 데이터 전송에 적합

</details>

## Q15. CPU 스케줄링 알고리즘을 설명해주세요. ⭐⭐⭐

<details>
<summary>답변 보기</summary>

### 핵심 답변
CPU 스케줄링은 준비 큐에 있는 프로세스 중 **어떤 프로세스에게 CPU를 할당할지** 결정하는 알고리즘입니다. 선점형(Preemptive)과 비선점형(Non-preemptive)으로 구분됩니다.

### 선점형 vs 비선점형

| 구분 | 비선점형 | 선점형 |
|------|---------|--------|
| 설명 | 프로세스가 자발적으로 CPU 반납 | OS가 강제로 CPU 회수 가능 |
| 응답성 | 낮음 | 높음 |
| 오버헤드 | 낮음 | 높음 (컨텍스트 스위칭) |
| 알고리즘 | FCFS, SJF, HRN | Round Robin, Priority(선점), MLFQ |

### 주요 알고리즘

#### 1. FCFS (First Come First Served) — 비선점
```
도착 순서대로 처리
단점: 앞의 긴 작업이 뒤를 막음 (Convoy Effect)

작업: P1(24ms), P2(3ms), P3(3ms) 순서로 도착
대기 시간: P1=0, P2=24, P3=27 → 평균 17ms
```

#### 2. SJF (Shortest Job First) — 비선점
```
남은 실행 시간이 가장 짧은 프로세스 우선
장점: 평균 대기 시간 최소화
단점: 긴 프로세스는 무한정 대기 가능 (Starvation)
      실제 실행 시간 예측 어려움
```

#### 3. HRN (Highest Response Ratio Next) — 비선점
```
우선순위 = (대기 시간 + 서비스 시간) / 서비스 시간

SJF의 Starvation 문제 보완
대기 시간이 길수록 우선순위 증가
```

#### 4. Round Robin (RR) — 선점
```
각 프로세스에 동일한 시간 할당량(Time Quantum) 부여
할당량 초과 시 다음 프로세스로 교체 (선점)

Time Quantum이 너무 크면 → FCFS와 유사
Time Quantum이 너무 작으면 → 컨텍스트 스위칭 오버헤드 증가

적정 Time Quantum: CPU 연산 시간의 80%를 처리할 수 있는 크기
```

#### 5. Priority Scheduling — 선점/비선점
```
우선순위가 높은 프로세스 먼저 실행
단점: 낮은 우선순위 프로세스의 Starvation
해결: Aging (대기할수록 우선순위 증가)
```

#### 6. 다단계 피드백 큐 (MLFQ, Multilevel Feedback Queue) — 선점
```
여러 개의 Ready Queue (높은 우선순위 ~ 낮은 우선순위)
새 프로세스는 가장 높은 우선순위 큐에 배치
할당량 내 완료 못하면 → 낮은 우선순위 큐로 이동

장점: I/O 위주 프로세스(짧은 CPU 버스트)가 높은 우선순위 유지
      CPU 위주 프로세스는 점점 낮은 큐로 이동
→ Linux의 CFS(Completely Fair Scheduler)도 이 개념 기반
```

### 성능 지표

| 지표 | 설명 |
|------|------|
| CPU 이용률 | CPU가 바쁜 시간 비율 |
| 처리량(Throughput) | 단위 시간당 완료 프로세스 수 |
| 반환 시간(Turnaround) | 제출~완료 시간 |
| 대기 시간(Waiting) | Ready Queue에서 기다린 시간 |
| 응답 시간(Response) | 첫 번째 응답까지 걸린 시간 |

### 면접관이 주목하는 포인트
- Round Robin의 Time Quantum 크기가 성능에 미치는 영향
- Starvation 문제와 Aging 해결 방법

### 꼬리 질문 대비
- "실제 OS에서는 어떤 알고리즘을 사용하나요?" → Linux는 CFS(완전 공정 스케줄러), Windows는 Priority Queue 기반 멀티레벨 피드백

</details>

## Q16. IPC(프로세스 간 통신)란 무엇인가요? ⭐⭐

<details>
<summary>답변 보기</summary>

### 핵심 답변
IPC(Inter-Process Communication)는 **서로 다른 프로세스가 데이터를 주고받는 메커니즘**입니다. 프로세스는 각자 독립된 메모리 공간을 가지므로 별도의 통신 방식이 필요합니다.

### IPC 방식 비교

| 방식 | 속도 | 데이터 크기 | 방향 | 특징 |
|------|------|-----------|------|------|
| Pipe | 보통 | 제한적 | 단방향 | 부모-자식 프로세스 간 |
| Named Pipe (FIFO) | 보통 | 제한적 | 단방향 | 무관한 프로세스 간 가능 |
| Message Queue | 보통 | 제한적 | 양방향 | 메시지 단위, 비동기 |
| Shared Memory | 빠름 | 대용량 | 양방향 | 동기화 필요 (세마포어) |
| Socket | 느림 | 제한 없음 | 양방향 | 네트워크 통신 포함 |
| Signal | - | 없음 | 단방향 | 이벤트 알림 전용 |

### 상세 설명

#### 1. Pipe
```
$ ls | grep ".txt"   ← 파이프(|)로 두 프로세스 연결

- 익명 파이프: 부모-자식 프로세스 간 단방향 통신
- 커널 버퍼를 통한 FIFO 데이터 전달
- 양방향은 파이프 2개 사용
```

#### 2. Shared Memory (공유 메모리)
```python
# 가장 빠른 IPC
# 여러 프로세스가 동일 메모리 영역에 직접 접근

import multiprocessing

shared = multiprocessing.Value('i', 0)  # 공유 정수

# 주의: 동시 접근 시 Race Condition 발생
# → 세마포어나 뮤텍스로 동기화 필수
with shared.get_lock():
    shared.value += 1
```

#### 3. Message Queue
```
- 커널이 관리하는 메시지 큐에 데이터 저장
- 비동기 통신: 수신자가 없어도 메시지 큐에 보관
- 메시지 타입으로 선택적 수신 가능

예: RabbitMQ, Kafka (분산 시스템의 MQ)
```

#### 4. Socket
```
- IPC 중 유일하게 네트워크를 통한 원격 통신 지원
- 로컬 소켓(Unix Domain Socket): 네트워크 없이 빠른 로컬 통신
- TCP/UDP 소켓: 다른 호스트와 통신

예: 웹 서버-데이터베이스 통신
```

### 선택 기준

```
대용량 + 고속 → Shared Memory (+ 세마포어)
비동기 메시징 → Message Queue
네트워크 통신 → Socket
단순 부모-자식 → Pipe
이벤트 알림 → Signal
```

### 면접관이 주목하는 포인트
- 공유 메모리가 빠른 이유 (복사 없이 메모리 직접 접근)
- 공유 메모리 사용 시 동기화 필요성

### 꼬리 질문 대비
- "스레드 간 통신과 프로세스 간 통신의 차이?" → 스레드는 메모리를 공유하므로 전역 변수로 직접 통신, 프로세스는 독립 메모리라 IPC 필요

</details>

## Q17. 파일 시스템이란 무엇인가요? ⭐⭐

<details>
<summary>답변 보기</summary>

### 핵심 답변
파일 시스템은 **OS가 저장 장치에서 데이터를 저장, 구성, 검색하는 방법**을 정의하는 구조입니다. 파일과 디렉토리를 관리하며 메타데이터와 실제 데이터 영역으로 나뉩니다.

### 파일 시스템 구성

```
┌──────────────────────────────────────┐
│             메타 영역                 │
│  - 수퍼블록: 파일 시스템 전체 정보     │
│  - i-node: 파일 속성 (권한, 크기, 시간) │
│  - 디렉토리: 파일명 ↔ i-node 번호 매핑 │
├──────────────────────────────────────┤
│             데이터 영역               │
│  - 실제 파일 내용                     │
└──────────────────────────────────────┘
```

### 파일 접근 방법

| 방법 | 설명 | 사용 사례 |
|------|------|---------|
| 순차 접근 (Sequential) | 처음부터 순서대로 읽기 | 텍스트 파일, 로그 파일 |
| 직접 접근 (Direct/Random) | 임의 위치에 바로 접근 | 데이터베이스, 인덱스 |
| 인덱스 접근 (Indexed) | 인덱스 테이블로 위치 파악 후 접근 | 대용량 파일, DB |

### 디렉토리 구조

| 구조 | 설명 | 장단점 |
|------|------|--------|
| 1단계 (Single Level) | 모든 파일이 하나의 디렉토리 | 단순하지만 이름 충돌 |
| 2단계 (Two Level) | 사용자별 디렉토리 | 사용자 간 격리 |
| 트리 구조 (Tree) | 계층적 디렉토리 (현재 표준) | 체계적, 경로 표현 |
| 그래프 구조 (Graph) | 공유 디렉토리 허용 (심볼릭 링크) | 유연하지만 순환 가능 |

### 주요 파일 시스템 종류

| OS | 파일 시스템 |
|----|-----------|
| Linux | ext4, XFS, Btrfs |
| Windows | NTFS, FAT32, exFAT |
| macOS | APFS, HFS+ |
| 네트워크 | NFS, SMB |

### i-node (Index Node)

```
파일 하나당 i-node 1개:
  - 파일 타입 (일반, 디렉토리, 링크)
  - 권한 (rwxrwxrwx)
  - 소유자, 그룹
  - 파일 크기
  - 타임스탬프 (생성, 수정, 접근)
  - 데이터 블록 포인터

ls -i 명령어로 i-node 번호 확인 가능
```

### 면접관이 주목하는 포인트
- i-node와 디렉토리의 관계 (파일명은 디렉토리에, 속성은 i-node에)
- 하드 링크 vs 심볼릭 링크 차이 (i-node 공유 vs 경로 참조)

### 꼬리 질문 대비
- "하드 링크와 심볼릭 링크의 차이?" → 하드 링크는 같은 i-node를 가리켜 파일 삭제 후에도 유효, 심볼릭 링크는 경로를 가리켜 원본 삭제 시 깨짐

</details>

## Q18. 페이징(Paging)과 세그멘테이션(Segmentation)의 차이는? ⭐⭐

<details>
<summary>답변 보기</summary>

### 핵심 답변
두 방식 모두 **가상 메모리를 물리 메모리에 매핑**하는 기법입니다. 페이징은 **고정 크기** 단위로, 세그멘테이션은 **가변 크기**(논리적 단위)로 분할합니다.

### 페이징 (Paging)

```
가상 메모리를 고정 크기 "페이지(Page)"로 분할
물리 메모리를 동일 크기 "프레임(Frame)"으로 분할

가상 주소 = [페이지 번호 | 오프셋]
물리 주소 = [프레임 번호 | 오프셋]

페이지 테이블: 페이지 번호 → 프레임 번호 매핑

장점:
  - 외부 단편화(External Fragmentation) 없음
  - 연속되지 않은 메모리에 프로세스 배치 가능

단점:
  - 내부 단편화(Internal Fragmentation) 발생
    (마지막 페이지가 꽉 차지 않으면 낭비)
  - 페이지 테이블 오버헤드
```

### 세그멘테이션 (Segmentation)

```
프로세스를 논리적 단위 "세그먼트"로 분할
세그먼트: 코드, 스택, 힙, 데이터 등

가상 주소 = [세그먼트 번호 | 오프셋]
물리 주소 = 세그먼트 기준 주소 + 오프셋

세그먼트 테이블: 번호 → (기준 주소, 한계값)

장점:
  - 내부 단편화 없음 (가변 크기라 딱 맞게 배치)
  - 논리적 단위로 보호(protection) 설정 가능
  - 공유 용이 (코드 세그먼트 공유)

단점:
  - 외부 단편화(External Fragmentation) 발생
    (가변 크기라 중간에 빈 공간 생김)
  - 연속 메모리 필요
```

### 비교

| 구분 | 페이징 | 세그멘테이션 |
|------|--------|------------|
| 분할 단위 | 고정 크기 (4KB 등) | 가변 크기 (논리 단위) |
| 내부 단편화 | 발생 | 없음 |
| 외부 단편화 | 없음 | 발생 |
| 주소 변환 | 페이지 테이블 | 세그먼트 테이블 |
| 공유 | 페이지 단위 공유 | 세그먼트 단위 공유 |
| 보호 | 페이지 단위 | 세그먼트 단위 (더 자연스러움) |

### 단편화 비교

```
내부 단편화 (Internal Fragmentation):
  할당된 공간 내부에 낭비 발생
  예: 4KB 페이지에 3.5KB 데이터 → 0.5KB 낭비

외부 단편화 (External Fragmentation):
  여러 곳에 흩어진 빈 공간 때문에 큰 연속 공간 없음
  예: 세그먼트 할당/해제 반복 시 중간에 구멍 발생
```

### 현대 OS: 세그먼테이션 + 페이징 혼합

```
x86: 세그먼테이션 → 페이징 2단계 주소 변환
Linux: 세그먼트를 사실상 비활성화, 페이징만 사용
→ 실제로는 페이징이 표준
```

### 면접관이 주목하는 포인트
- 내부/외부 단편화의 차이와 각 방식이 어떤 단편화를 유발하는지
- 현대 OS가 페이징을 주로 사용하는 이유

### 꼬리 질문 대비
- "Demand Paging이란?" → 필요한 페이지만 메모리에 올리고 나머지는 디스크에 두는 방식, 가상 메모리의 핵심 기법
