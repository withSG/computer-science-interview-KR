## Q1. CPU의 구성요소와 명령어 실행 사이클을 설명해주세요. ⭐⭐⭐

<details>
<summary>답변 보기</summary>

### 핵심 답변
CPU는 **ALU(산술논리장치)**, **제어장치(CU)**, **레지스터**로 구성되며, 명령어를 Fetch → Decode → Execute 사이클로 처리합니다.

### CPU 구성요소

| 구성요소 | 역할 |
|---------|------|
| ALU | 산술(+, -, ×, ÷)·논리(AND, OR, NOT) 연산 수행 |
| 제어장치(CU) | 명령어 해석 및 각 장치 제어 신호 생성 |
| 레지스터 | CPU 내부의 초고속 임시 저장소 |
| 버스 | 데이터·주소·제어 신호 전송 경로 |

### 주요 레지스터

| 레지스터 | 약칭 | 역할 |
|---------|------|------|
| Program Counter | PC | 다음 실행할 명령어 주소 저장 |
| Memory Address Register | MAR | 메모리 접근 주소 저장 |
| Memory Buffer Register | MBR | 메모리에서 읽거나 쓸 데이터 임시 저장 |
| Instruction Register | IR | 현재 실행 중인 명령어 저장 |
| Accumulator | ACC | 연산 결과 저장 |

### 명령어 실행 사이클 (Instruction Cycle)

```
① Fetch (인출)
   PC → MAR → 메모리에서 명령어 읽기 → MBR → IR
   PC = PC + 1 (다음 명령어 주소로 증가)

② Decode (해석)
   IR의 명령어를 제어장치가 해석
   어떤 연산인지, 피연산자는 무엇인지 파악

③ Execute (실행)
   ALU가 연산 수행 또는 메모리 접근
   결과를 레지스터나 메모리에 저장

④ (필요 시) Memory Access / Write Back
```

### 명령어 예시

```
ADD R1, R2, R3   → R1 = R2 + R3  (ALU 연산)
LOAD R1, 100     → R1 = MEM[100] (메모리 읽기)
STORE R1, 200    → MEM[200] = R1 (메모리 쓰기)
JUMP 300         → PC = 300      (분기)
```

### 면접관이 주목하는 포인트
- PC(Program Counter)의 역할과 증가 시점
- Fetch 단계에서 메모리 접근이 왜 느린지 (캐시의 필요성)

### 꼬리 질문 대비
- "파이프라이닝이란?" → 여러 명령어의 Fetch/Decode/Execute를 동시에 처리해 처리량을 높이는 기법

</details>

## Q2. 캐시 메모리란 무엇이며, 어떻게 동작하나요? ⭐⭐⭐

<details>
<summary>답변 보기</summary>

### 핵심 답변
캐시 메모리는 CPU와 메인 메모리(DRAM) 사이의 속도 차이를 줄이기 위한 **고속 소용량 메모리**입니다. **지역성 원리**를 활용해 자주 사용하는 데이터를 미리 저장합니다.

### 캐시 계층 구조

| 계층 | 속도 | 크기 | 위치 |
|------|------|------|------|
| L1 캐시 | ~1ns | 32~64KB | 코어 내부 |
| L2 캐시 | ~4ns | 256KB~1MB | 코어 내부/외부 |
| L3 캐시 | ~10ns | 4~32MB | 여러 코어 공유 |
| DRAM | ~100ns | GB 단위 | 메인보드 |

### 지역성 원리 (Locality)

```
시간적 지역성 (Temporal Locality)
  → 최근에 접근한 데이터를 곧 다시 접근할 가능성이 높음
  → 예: 루프 내의 변수

공간적 지역성 (Spatial Locality)
  → 접근한 주소 근처의 데이터를 곧 접근할 가능성이 높음
  → 예: 배열의 순차 접근
```

### 캐시 동작 방식

```
① CPU가 데이터 요청
② 캐시에 데이터가 있으면 → Cache Hit → 빠르게 반환
③ 캐시에 없으면 → Cache Miss → 메인 메모리에서 읽어 캐시에 저장 후 반환
```

### 캐시 미스(Cache Miss) 3종류

| 종류 | 원인 |
|------|------|
| Cold Miss | 처음 접근하는 데이터 (강제 미스) |
| Conflict Miss | 다른 데이터가 같은 캐시 라인에 매핑되어 교체 발생 |
| Capacity Miss | 캐시 용량 부족으로 필요한 데이터가 없음 |

### 캐시 매핑 방식

| 방식 | 설명 | 장점 | 단점 |
|------|------|------|------|
| 직접 매핑 | 메모리 블록이 특정 캐시 라인에 고정 | 구현 간단 | Conflict Miss 빈번 |
| 완전 연관 매핑 | 임의 캐시 라인에 저장 | Conflict Miss 없음 | 검색 느림, 비용 높음 |
| 집합 연관 매핑 | N-way: N개 후보 중 저장 | 절충안 (실제 사용) | 복잡도 중간 |

### 면접관이 주목하는 포인트
- 캐시가 필요한 이유 (CPU-DRAM 속도 격차)
- 지역성 원리와 캐시 효율성의 관계

### 꼬리 질문 대비
- "캐시 일관성 문제란?" → 멀티코어에서 각 코어의 L1 캐시가 같은 메모리를 다르게 캐싱할 때 발생, MESI 프로토콜로 해결

</details>

## Q3. 고정 소수점과 부동 소수점의 차이는 무엇인가요? ⭐⭐

<details>
<summary>답변 보기</summary>

### 핵심 답변
**고정 소수점**은 소수점 위치가 고정되어 정밀도는 일정하지만 표현 범위가 좁고, **부동 소수점**은 소수점이 움직여 넓은 범위를 표현하지만 정밀도 오차가 발생할 수 있습니다.

### 고정 소수점 (Fixed Point)

```
구조: [부호 비트 | 정수부 | 소수부]
예: 16비트 기준 → 부호(1) + 정수(7) + 소수(8)

장점: 연산 간단, 속도 빠름, 오차 없음
단점: 표현 범위 좁음, 매우 크거나 작은 수 표현 불가

사용: 임베디드 시스템, 실시간 제어 시스템, DSP
```

### 부동 소수점 (Floating Point, IEEE 754)

```
구조: [부호 비트 | 지수부(Exponent) | 가수부(Mantissa)]

단정도(32비트): 부호(1) + 지수(8) + 가수(23)
배정도(64비트): 부호(1) + 지수(11) + 가수(52)

값 = (-1)^부호 × 1.가수 × 2^(지수 - bias)

장점: 매우 넓은 표현 범위 (±3.4 × 10^38)
단점: 정밀도 제한 (0.1 + 0.2 ≠ 0.3 문제)

사용: 과학 계산, 그래픽, 일반 프로그래밍
```

### 비교

| 구분 | 고정 소수점 | 부동 소수점 |
|------|-----------|-----------|
| 소수점 위치 | 고정 | 이동 가능 |
| 표현 범위 | 좁음 | 넓음 |
| 정밀도 | 일정 | 오차 발생 가능 |
| 연산 속도 | 빠름 | 상대적으로 느림 |
| 사용 분야 | 임베디드, DSP | 일반 프로그래밍 |

### 부동 소수점 오차 예시

```python
>>> 0.1 + 0.2
0.30000000000000004  # 정확히 0.3이 아님

# 해결: 금융 계산에는 BigDecimal/Decimal 사용
```

### 면접관이 주목하는 포인트
- 부동 소수점 오차가 발생하는 이유 (이진수로 정확히 표현 불가)
- 금융 시스템에서 부동 소수점을 쓰면 안 되는 이유

### 꼬리 질문 대비
- "0.1 + 0.2 != 0.3인 이유는?" → 0.1, 0.2를 이진수로 변환하면 무한 소수가 되어 가수부에서 잘리기 때문

</details>

## Q4. 패리티 비트와 해밍 코드(ECC)란 무엇인가요? ⭐⭐

<details>
<summary>답변 보기</summary>

### 핵심 답변
**패리티 비트**는 1비트 오류를 **검출**하는 가장 간단한 오류 검출 방식이고, **해밍 코드(ECC)**는 오류를 **검출하고 수정**할 수 있는 방식입니다.

### 패리티 비트 (Parity Bit)

```
짝수 패리티: 전체 1의 개수가 짝수가 되도록 패리티 비트 설정
홀수 패리티: 전체 1의 개수가 홀수가 되도록 패리티 비트 설정

예시 (짝수 패리티):
데이터: 1011001 → 1의 개수 = 4개 (짝수)
패리티 비트: 0 → 전송: 1011001 0

데이터: 1011000 → 1의 개수 = 3개 (홀수)
패리티 비트: 1 → 전송: 1011000 1
```

```
한계:
  ✓ 1비트 오류 검출 가능
  ✗ 2비트 동시 오류는 검출 불가
  ✗ 오류 위치 파악 불가 (수정 불가)
```

### 해밍 코드 (Hamming Code / ECC)

```
원리: 여러 패리티 비트를 전략적으로 배치하여 오류 위치 특정

패리티 비트 위치: 2의 거듭제곱 위치 (1, 2, 4, 8, ...)

예시: 4비트 데이터 1011 전송 시
위치:  1  2  3  4  5  6  7
내용:  P1 P2 d1 P4 d2 d3 d4
       P1 P2  1  P4  0  1  1

P1 = 위치 1,3,5,7의 XOR = 1⊕0⊕1 = 0
P2 = 위치 2,3,6,7의 XOR = 1⊕1⊕1 = 1
P4 = 위치 4,5,6,7의 XOR = 0⊕1⊕1 = 0

수신 측에서 패리티 검사 결과가 P4=1, P2=1, P1=0이면
오류 위치 = 4×1 + 2×1 + 1×0 = 6번 위치 → 수정
```

### 실제 사용

| 방식 | 용도 |
|------|------|
| 패리티 비트 | RAID 5, 기본 오류 검출 |
| ECC 메모리 | 서버용 RAM (1비트 수정, 2비트 검출) |
| CRC | 네트워크, 파일 전송 오류 검출 |

### 면접관이 주목하는 포인트
- 패리티와 ECC의 차이 (검출 vs 검출+수정)
- ECC 메모리가 서버에서 쓰이는 이유

### 꼬리 질문 대비
- "RAID 5에서 패리티를 어떻게 활용하나요?" → 디스크 1개 장애 시 나머지 디스크의 데이터 + 패리티로 복구

</details>

## Q5. ARM 프로세서의 특징과 x86과의 차이점은? ⭐

<details>
<summary>답변 보기</summary>

### 핵심 답변
ARM은 **RISC(Reduced Instruction Set Computer)** 아키텍처로 단순한 명령어 세트와 낮은 전력 소비가 특징이고, x86은 **CISC(Complex Instruction Set Computer)** 아키텍처로 복잡한 명령어를 지원해 높은 성능을 냅니다.

### RISC vs CISC 비교

| 구분 | RISC (ARM) | CISC (x86) |
|------|-----------|-----------|
| 명령어 수 | 적음 (단순) | 많음 (복잡) |
| 명령어 길이 | 고정 (4바이트) | 가변 |
| 실행 사이클 | 1 사이클 | 다수 사이클 |
| 전력 소비 | 낮음 | 높음 |
| 발열 | 낮음 | 높음 |
| 성능 | 스칼라 연산 우수 | 복잡 연산 우수 |
| 사용처 | 모바일, IoT, 서버(최근) | PC, 서버(전통) |

### ARM의 부상

```
모바일 시대:
  → 배터리 수명이 중요 → 낮은 전력의 ARM 채택
  → Apple A 시리즈, Qualcomm Snapdragon 등

서버/데스크탑 침투:
  → Apple M1/M2 (ARM 기반) → x86 대비 성능/와트 압도
  → AWS Graviton (ARM 기반 서버)
  → Microsoft Windows on ARM
```

### 소프트웨어 호환성

```
x86 → ARM 이식 시 고려 사항:
  - 엔디언(Byte Order): ARM은 bi-endian (기본 little-endian)
  - 정렬(Alignment): ARM은 정렬 위반 시 오류 가능
  - 컴파일러 재빌드 필요
  
해결:
  - Apple Rosetta 2: x86 바이너리를 ARM에서 동적 번역
```

### 면접관이 주목하는 포인트
- RISC와 CISC의 근본적 설계 철학 차이
- ARM이 모바일에서 지배적인 이유 (전력 효율)

### 꼬리 질문 대비
- "Apple이 Intel에서 ARM으로 전환한 이유는?" → 전력 효율과 성능을 동시에 향상, M1이 x86 대비 성능/와트 압도적 우위

</details>
