# 행동 면접 (Behavioral Interview) 질문 & 답변

## 사용 방법
1. 질문을 먼저 읽고 스스로 답변해보세요
2. STAR 기법으로 구조화해서 답변하세요
3. 실제 경험을 기반으로 구체적인 숫자와 결과를 포함하세요

---

## STAR 기법

```
S - Situation (상황)
    "어떤 프로젝트/상황이었는지 배경 설명"

T - Task (과제)
    "당신의 역할과 해결해야 할 문제"

A - Action (행동)
    "구체적으로 어떤 행동을 취했는지"

R - Result (결과)
    "정량적 결과와 배운 점"
```

## Q1. 가장 도전적이었던 기술적 문제는? ⭐⭐⭐

<details>
<summary>답변 예시</summary>

### STAR 답변 구조

**Situation (상황)**
```
"이커머스 서비스에서 블랙프라이데이 프로모션을 앞두고 있었습니다.
평소 대비 10배의 트래픽이 예상되었고,
기존 시스템은 3배 트래픽에서도 장애가 발생한 이력이 있었습니다."
```

**Task (과제)**
```
"백엔드 팀 리드로서 2주 안에 시스템이
10배 트래픽을 버틸 수 있도록 개선해야 했습니다."
```

**Action (행동)**
```
"1. 부하 테스트로 병목 구간 파악
   - 상품 조회 API: DB 쿼리 지연
   - 장바구니: Redis 연결 풀 고갈
   - 결제: 동기 처리로 인한 타임아웃

2. 각 문제별 해결책 적용
   - 상품: Redis 캐시 + Read Replica
   - 장바구니: 커넥션 풀 확대 + 클러스터링
   - 결제: 비동기 처리 + 재시도 로직

3. 인프라 확장
   - K8s HPA 설정 (CPU 70% → 스케일아웃)
   - Auto Scaling 최대치 상향"
```

**Result (결과)**
```
"블랙프라이데이 당일:
- 평소 대비 12배 트래픽 처리 성공
- 평균 응답시간 200ms 유지 (목표 500ms)
- 장애 없이 매출 전년 대비 150% 달성

이 경험으로 사전 부하 테스트와
점진적 배포의 중요성을 깊이 깨달았습니다."
```

### 면접관이 주목하는 포인트
- 문제 정의의 명확성
- 체계적인 접근 방법
- 정량적 결과
- 배운 점/성장

</details>

## Q2. 팀원과 의견 충돌이 있었던 경험은? ⭐⭐⭐

<details>
<summary>답변 예시</summary>

### STAR 답변 구조

**Situation (상황)**
```
"신규 서비스의 데이터베이스 선택 과정에서
시니어 개발자와 의견이 달랐습니다.
시니어는 PostgreSQL, 저는 MongoDB를 선호했습니다."
```

**Task (과제)**
```
"팀의 기술 선택에 영향을 주는 중요한 결정이었고,
합리적인 방식으로 합의점을 찾아야 했습니다."
```

**Action (행동)**
```
"1. 객관적 비교 자료 준비
   - 요구사항 체크리스트 작성
   - 각 DB의 장단점 정리
   - 유사 사례 리서치

2. 기술 토론 세션 제안
   - 30분 미팅 설정
   - 각자 입장 발표
   - 질의응답

3. 핵심 쟁점 파악
   - 시니어의 우려: 트랜잭션 일관성
   - 저의 관점: 스키마 유연성

4. PoC 제안
   - 양쪽 DB로 프로토타입 구현
   - 실제 성능 비교"
```

**Result (결과)**
```
"PoC 결과 PostgreSQL이 우리 요구사항에 더 적합했습니다.
트랜잭션이 많은 주문 도메인 특성상
ACID가 중요했기 때문입니다.

제 의견이 채택되지 않았지만,
데이터 기반 의사결정의 중요성을 배웠고
시니어와의 관계도 더 좋아졌습니다."
```

### 면접관이 주목하는 포인트
- 갈등 해결 방식 (감정 vs 논리)
- 열린 마음으로 다른 의견 수용
- 최종 결정에 대한 태도

</details>

## Q3. 실패했던 프로젝트 경험이 있나요? ⭐⭐⭐

<details>
<summary>답변 예시</summary>

### STAR 답변 구조

**Situation (상황)**
```
"사내 레거시 시스템을 MSA로 전환하는
6개월 프로젝트를 리드했습니다."
```

**Task (과제)**
```
"10년 된 모놀리식 시스템을 마이크로서비스로 분리하고
다운타임 없이 마이그레이션하는 것이 목표였습니다."
```

**Action (행동)**
```
"1. 처음 3개월
   - 도메인 분석 후 8개 서비스로 분리 설계
   - 각 서비스 개발 시작

2. 문제 발생
   - 서비스 간 의존성 복잡도 과소평가
   - 통합 테스트 환경 부재
   - 팀원들의 MSA 경험 부족

3. 대응 시도
   - 서비스 수 축소 (8개 → 4개)
   - 스터디 세션 진행
   - 일정 재협상 시도"
```

**Result (결과)**
```
"결국 프로젝트는 6개월 → 10개월로 지연되었고,
4개 서비스 중 2개만 분리 완료되었습니다.

실패 원인 분석:
1. 충분한 PoC 없이 전면 전환 시도
2. 팀 역량 고려 부족
3. 단계적 마이그레이션 계획 부재

배운 점:
- Strangler Fig 패턴으로 점진적 전환
- 팀 교육에 충분한 시간 투자
- 기술 도입 전 PoC 필수"
```

### 면접관이 주목하는 포인트
- 실패를 인정하는 태도
- 원인 분석의 깊이
- 구체적인 교훈과 성장

</details>

## Q4. 일정이 촉박한 상황을 어떻게 처리했나요? ⭐⭐

<details>
<summary>답변 예시</summary>

### STAR 답변 구조

**Situation (상황)**
```
"고객사 요청으로 새로운 결제 연동을
2주 안에 완료해야 했습니다.
일반적으로 4주가 필요한 작업이었습니다."
```

**Task (과제)**
```
"비즈니스적으로 중요한 계약이었고,
품질을 유지하면서 일정을 맞춰야 했습니다."
```

**Action (행동)**
```
"1. 범위 협상
   - 필수 기능과 Nice-to-have 분리
   - 1차 배포: 핵심 결제만
   - 2차 배포: 부가 기능

2. 리소스 조정
   - 다른 작업 일시 중단
   - 팀원 2명 추가 지원 요청

3. 효율화
   - 코드 리뷰 간소화 (페어 프로그래밍 대체)
   - 기존 라이브러리 최대 활용
   - 야근은 주 2회로 제한 (지속가능성)

4. 리스크 관리
   - 매일 15분 스탠드업
   - 블로커 즉시 공유
   - 일정 위험 시 조기 경보"
```

**Result (결과)**
```
"2주 안에 핵심 기능 배포 완료.
부가 기능은 이후 1주 추가로 완료.

계약 성사 + 고객사 만족
팀원 번아웃 없이 마무리

배운 점:
- 초기 범위 협상의 중요성
- 지속가능한 페이스 유지
- 투명한 커뮤니케이션"
```

</details>

## Q5. 기술 부채를 어떻게 관리했나요? ⭐⭐

<details>
<summary>답변 예시</summary>

### STAR 답변 구조

**Situation (상황)**
```
"빠른 성장 과정에서 쌓인 기술 부채로
새 기능 개발 속도가 50% 이상 저하되었습니다.
테스트 커버리지 20%, 중복 코드 다수."
```

**Task (과제)**
```
"신규 기능 개발을 유지하면서
점진적으로 기술 부채를 해소해야 했습니다."
```

**Action (행동)**
```
"1. 기술 부채 목록화
   - 팀원들과 브레인스토밍
   - 영향도/노력 매트릭스 작성
   - 우선순위 결정

2. 20% 룰 도입
   - 스프린트의 20%를 리팩토링에 할당
   - '보이스카우트 룰' 적용

3. 점진적 개선
   - 테스트 커버리지 주 2% 증가 목표
   - 새 코드는 반드시 테스트 포함
   - 기존 코드 수정 시 리팩토링 병행

4. 자동화
   - SonarQube 도입
   - PR 품질 게이트 설정"
```

**Result (결과)**
```
"6개월 후:
- 테스트 커버리지: 20% → 65%
- 코드 중복: 40% 감소
- 버그 발생률: 30% 감소
- 기능 개발 속도: 정상화

기술 부채는 '나중에 한번에' 가 아니라
'지속적으로 조금씩' 해결하는 것이 효과적임을 배웠습니다."
```

</details>

## Q6. 후배/신입 개발자를 어떻게 온보딩시켰나요? ⭐⭐

<details>
<summary>답변 예시</summary>

### STAR 답변 구조

**Situation (상황)**
```
"팀에 신입 개발자 2명이 합류했습니다.
기존 온보딩 문서가 없어 적응에 2-3개월이 걸렸습니다."
```

**Task (과제)**
```
"신입 개발자가 1개월 안에
독립적으로 업무를 수행할 수 있도록 지원해야 했습니다."
```

**Action (행동)**
```
"1. 온보딩 문서 체계화
   - 로컬 환경 설정 가이드
   - 아키텍처 다이어그램
   - 주요 코드 설명
   - FAQ 정리

2. 버디 프로그램
   - 시니어 1:1 매칭
   - 매일 30분 질의응답 시간
   - 심리적 안전감 조성

3. 단계별 과제 부여
   - Week 1: 환경 설정 + 간단한 버그 수정
   - Week 2: 작은 기능 개발
   - Week 3: 코드 리뷰 참여
   - Week 4: 독립적 태스크

4. 피드백 루프
   - 주간 1:1 미팅
   - 막히는 부분 즉시 해결
   - 성장 포인트 공유"
```

**Result (결과)**
```
"2명 모두 4주차에 독립적 업무 수행.
온보딩 만족도 조사: 9/10점.

이후 온보딩 가이드가 팀 표준이 되어
신규 입사자 적응 기간 50% 단축.

가르치면서 저도 기술을 더 깊이 이해하게 되었습니다."
```

</details>

## Q7. 왜 이 회사에 지원했나요? ⭐⭐⭐

<details>
<summary>답변 가이드</summary>

### 답변 구조

```
1. 회사에 대한 이해 (리서치)
   - 비즈니스 모델
   - 기술 스택
   - 최근 뉴스/성과

2. 나의 강점과 연결
   - 회사 니즈 + 나의 경험
   - 기여할 수 있는 부분

3. 성장 관점
   - 이 회사에서 배우고 싶은 것
   - 커리어 목표와의 연결
```

### 답변 예시

```
"세 가지 이유로 지원했습니다.

첫째, 기술적 도전.
[회사명]은 일 1억 트랜잭션을 처리하는 시스템을 운영하고 있습니다.
이런 대규모 시스템을 경험하고 싶었습니다.

둘째, 기술 문화.
기술 블로그와 컨퍼런스 발표를 보며
엔지니어링 문화가 인상적이었습니다.
특히 [특정 기술/프로젝트]에 대한 글이 기억에 남습니다.

셋째, 도메인 관심.
[도메인]에 관심이 많았고,
[회사명]이 이 분야의 리더라고 생각합니다.

저의 클라우드 인프라 경험과
대용량 트래픽 처리 경험이
[팀명]에 기여할 수 있다고 생각합니다."
```

### 피해야 할 답변
```
❌ "대기업이라서요"
❌ "연봉이 높아서요"
❌ "집에서 가까워서요"
❌ (회사에 대해 아는 게 없는 것이 드러남)
```

</details>

## 준비 팁

### 경험 정리 체크리스트

```
☐ 도전적인 기술 문제 해결 경험 2-3개
☐ 팀 협업/갈등 해결 경험 1-2개
☐ 실패하고 배운 경험 1개
☐ 리더십/멘토링 경험 1개
☐ 시간 압박 상황 대처 경험 1개
☐ 코드/프로세스 개선 경험 1개
```

### STAR 답변 연습

```
1. 각 경험을 STAR 형식으로 정리
2. 2분 내외로 요약 연습
3. 후속 질문 예상하고 준비
4. 정량적 결과 반드시 포함
```

### 면접 당일 팁

```
- 구체적인 숫자와 결과 언급
- "우리"보다 "제가" 사용 (본인 역할 명확히)
- 기술적 디테일과 비즈니스 임팩트 균형
- 솔직하게, 그러나 긍정적으로
```
